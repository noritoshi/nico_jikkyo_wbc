# リリース作業

以下の手順を順番に実行してください。

## 0. 事前チェック（リリースすべきか判断）

- `git tag --sort=-v:refname | head -1` で最新のリリースタグを取得する
- `git log {最新タグ}..HEAD --oneline` で前回リリースからのコミット一覧を確認する
- `git diff {最新タグ}..HEAD --stat` で変更ファイルの概要を確認する

### 判断基準:
- **コミットが0件**: 「前回リリースから変更がありません。リリースを中止します。」と伝えて終了する
- **コミットが1〜2件で軽微な修正のみ**: 「変更が少ないですが、リリースしますか？」とユーザーに確認する。NOなら終了
- **それ以外**: リリースを続行する

## 1. 変更内容の把握とバージョン判断

変更内容を分析し、以下の基準でバージョン番号を決定する:

### バージョニング基準 (semver: MAJOR.MINOR.PATCH):
- **PATCH** (x.y.Z): バグ修正、軽微な調整、リファクタリング、テキスト修正
- **MINOR** (x.Y.0): 新機能追加、既存機能の大幅改善、新しいUIコンポーネント追加
- **MAJOR** (X.0.0): 破壊的変更、アーキテクチャの大幅変更、互換性のない変更

### 判断フロー:
1. 引数 `$ARGUMENTS` でバージョンが指定されていればそれを使う
2. 指定がなければ変更内容から PATCH か MINOR かを自動判断する
3. MAJOR に上げるべきと判断した場合は、理由を添えてユーザーに確認する（勝手に上げない）
4. 判断に迷う場合（PATCH か MINOR か微妙な場合）もユーザーに確認する

決定したバージョンと理由をユーザーに提示してから次に進む。

## 2. Knowledge ドキュメント更新

- `doc/knowledge/niconico_comment_protobuf.md` に、今回の変更に関連する技術仕様・設計判断があれば追記する
- 既存の記述と重複しないようにする
- 変更に関連がなければスキップ

## 3. README 更新

- まず `README.md` を読んで現在の内容を確認する
- `git diff {最新タグ}..HEAD` の変更内容に基づいて以下を更新する:
  - 新機能があれば「機能」セクションに追加する
  - 既存機能の変更・改善があれば該当箇所を更新する
  - 削除された機能があれば該当箇所を削除する
  - 設定方法・使い方に変更があれば反映する（例: 新しいAPI Key設定の追加など）
- 既存の記述と重複しないようにする
- 変更に関連がなければスキップ

## 4. デバッグフラグをOFFにする

- `src/background.js` の `const DEBUG = true;` を `const DEBUG = false;` に変更する
- `src/offscreen.js` の `const DEBUG = true;` を `const DEBUG = false;` に変更する

## 5. バージョンアップ

- `manifest.json` の `version` をステップ1で決定したバージョンに更新する

## 6. コミット

- 変更ファイルを `git add` してコミットする
- コミットメッセージ: `v{バージョン}: {変更の要約}`
- Co-Authored-By ヘッダーを含める

## 7. GitHub リリース作成

- `git tag v{バージョン}` でタグを作成
- `git push origin main --tags` でプッシュ
- プロジェクトルートを ZIP 化（`.claude/` ディレクトリは除外）:
  ```
  zip -r niko-jikkyo-v{バージョン}.zip . -x '.claude/*' -x '.git/*' -x '*.zip'
  ```
- `gh release create v{バージョン} niko-jikkyo-v{バージョン}.zip --title "v{バージョン}" --notes "{変更内容}"` でリリース作成
- ZIP ファイルを削除
